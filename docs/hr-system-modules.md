# HR 系统 — 模块列表（讨论稿）

> **目标：** 构建一个覆盖企业核心人力场景的 HR 管理系统，支持从招聘到离职的全生命周期管理。
> **状态：** 模块初稿，待讨论确认范围与优先级。

## 二、各模块详细说明

### 1. 组织架构管理

- **部门管理**：树形结构，支持多级部门，部门合并、拆分、撤销
- **岗位管理**：岗位定义、岗位说明书、岗位职级
- **编制管理**：各岗位编制数、在职数、空缺数，超编预警
- **汇报关系**：直线汇报 / 虚线汇报，组织图的自动生成
- **历史快照**：组织架构变更历史，支持指定日期回溯

### 2. 员工管理

- **员工类型管理**：正式员工 / 劳务派遣 / 业务外包 / 实习生 / 顾问 / 兼职工，不同类型对应不同的管理规则（合同模板、考勤方式、薪资公式等）
- **员工档案**：基本信息、联系方式、教育经历、工作经历、家庭成员
- **入职管理**：入职登记、材料上传、工号生成、入职引导
- **离职管理**：离职申请、交接清单、离职证明、离职分析
- **员工状态**：试用 / 正式 / 借调 / 离职 / 退休，状态流转记录
- **证件管理**：身份证、护照、劳动合同、学历证等到期提醒
- **黑名单**：离职原因标记，防止不当再录用

### 3. 考勤管理

- **排班管理**：固定班、轮班、弹性工时，排班日历
- **打卡记录**：多种打卡方式（GPS / WiFi / 人脸 / 指纹），异常标注
- **请假管理**：年假 / 事假 / 病假 / 婚假 / 产假 / 丧假等，额度管理
- **加班管理**：加班申请、工时折算、调休/加班费
- **出差管理**：出差申请、行程、费用关联
- **考勤统计**：迟到 / 早退 / 旷工 / 出勤率统计
- **特殊日历**：法定节假日、公司假日、调休日配置

### 4. 薪资管理

- **薪资结构**：基本工资 / 岗位工资 / 绩效工资 / 津贴 / 补贴，可配置公式
- **薪资核算**：按月考勤 + 绩效 + 社保扣款自动计算，支持批量导入导出
- **调薪管理**：调薪申请、生效日期、调薪历史
- **个税计算**：累计预扣法，专项附加扣除维护
- **工资单**：月工资单生成、员工自助查看、历史查询
- **银行报盘**：工资条转银行格式，批量发放
- **年终奖**：全年一次性奖金计算

### 5. 绩效管理

- **指标库**：KPI / OKR / 360 指标定义，可复用
- **绩效计划**：考核周期（月度 / 季度 / 年度）、目标设定、权重分配
- **绩效评估**：自评 + 上级评 + 同级评 + 下属评（360）
- **绩效面谈**：评估结果确认、改进计划、签名确认
- **结果应用**：绩效等级（S/A/B/C/D）、与薪资 / 晋升 / 培训联动
- **绩效申诉**：员工申诉通道、复核流程

### 6. 招聘管理

- **招聘需求**：用人部门发起、审批、编制校验
- **职位发布**：多平台一键发布（猎聘 / BOSS / 智联等）
- **简历管理**：简历解析、标签、评分、人才库
- **面试流程**：面试安排、面试评价表、面试官反馈
- **Offer 管理**：Offer 审批、发放、接受/拒绝
- **入职衔接**：Offer 确认 → 入职登记的数据传递
- **招聘看板**：渠道效果、转化率、招聘周期

### 7. 培训管理

- **课程管理**：线上 / 线下课程、讲师管理、课程资料
- **培训计划**：年度计划、季度计划、单次培训安排
- **培训实施**：报名、签到、考试、课程评价
- **学分体系**：岗位必修学分、年度学分统计
- **证书管理**：培训证书、职业资格证书到期提醒
- **培训档案**：员工培训记录、个人成长档案

### 8. 人才发展

- **继任计划**：关键岗位识别、候选人池、ready-now 评估
- **人才盘点**：九宫格、人才地图、高潜人才识别
- **职业路径**：双通道（管理 / 专业）、晋升条件、晋升申请
- **IDP**：个人发展计划制定与跟踪

### 9. 三方厂商管理（供应商管理）

> 管理为企业提供人力服务的第三方供应商（劳务派遣公司、外包服务商、猎头、培训机构等），从准入到结算的全生命周期。

- **供应商准入**：资质审核（营业执照、劳务派遣许可证、人力资源服务许可证）、行业经验、案例评估
- **供应商档案**：基本信息、联系人、银行账户、资质证件（到期提醒）、合作范围标签
- **合同管理**：框架协议、服务合同、保密协议，合同期限与续签预警
- **服务范围**：按供应商定义可提供的服务类型（派遣 / 外包 / 招聘 / 培训 / 后勤等）
- **供应商评级**：评分体系（响应速度 / 人员质量 / 合规性 / 价格竞争力），定期考评
- **供应商结算**：服务费用对账、发票管理、付款计划、结算记录
- **供应商门户**：供应商自助登录，提交人员名单、考勤数据、对账单、发票
- **黑白名单**：违约 / 纠纷记录，限制参与新项目

### 10. 三方员工管理（外包 / 派遣 / 实习）

> 管理非自有编制但实际在本企业工作的员工群体——外包人员、劳务派遣人员、实习生等。与正式员工共享部分管理流程，但用工关系归属供应商。

- **人员类型**：劳务派遣、业务外包、项目外包、实习生、退休返聘、顾问，各类型对应不同的合规规则
- **人员档案**：基本信息、所属供应商、服务岗位、入场日期、合同期限、到期提醒
- **入场管理**：供应商提交人员信息 → 企业审核 → 工卡/账号/权限开通 → 岗前培训
- **退场管理**：离职申请（通过供应商）、资产归还、权限回收、退场确认
- **考勤管理**：与正式员工统一打卡规则，或独立考勤方案（外包人员按项目工时）
- **工时与费用**：人员工时填报、加班申请、差旅报销，供供应商结算对账
- **供应商费用核对**：企业端确认工时 / 考勤 → 供应商开票 → 财务付款
- **转正通道**：外包转正式、实习转正式流程，编制校验，Offer 衔接
- **风险预警**：合同到期、资质过期、人员超期服务、供应商违规
- **统计报表**：三方人员分布（按供应商 / 部门 / 岗位）、费用汇总、人天统计

### 11. 员工自助与互动 (ESS / MSS)

> 企业内所有用户（员工、经理、HR）的统一门户，将各模块功能以角色化的工作台形式呈现。不承载独立的业务逻辑，而是将其他模块的能力聚合并按角色分发。

- **角色门户**：
  - **ESS（员工自助）**：我的档案、请假申请、加班申请、出差申请、工资单、绩效自评、培训报名、同事查询
  - **MSS（经理工作台）**：团队组织图、下属考勤、审批待办与批量处理、绩效评估（对下属）、招聘需求发起、团队薪资概览
  - **HR 工作台**：待办入职、合同到期提醒、审批中心、报表快捷入口、员工事务集中处理
- **个人中心**：个人资料编辑、密码修改、头像、消息偏好设置、登录日志
- **通知公告**：公司公告发布、制度文件推送、消息提醒（待办 / 审批 / 到期提醒）、已读/未读跟踪、弹窗强读
- **日程与待办**：个人待办列表聚合、审批待办、会议日历、生日 / 入职周年提醒
- **员工服务**：
  - **智能助手**：常见 HR 问题 FAQ、政策检索、办事指南
  - **证明开具**：在职证明、收入证明在线申请、电子签章、自助下载
  - **福利查询**：社保公积金缴纳明细、商业保险信息、体检预约
  - **意见反馈**：匿名 / 实名投诉建议、处理进度跟踪
- **同事互动**：企业通讯录（组织树 + 搜索）、同事生日祝福、团队活动通知、同事关怀（积分/点赞）
- **移动端适配**：响应式 H5 / 小程序 / 企微 / 钉钉 / 飞书集成，消息推送打通

### 12. 社保与公积金

- **城市策略**：不同城市的基数上下限、比例配置
- **缴纳计算**：个人扣款 + 企业承担明细
- **申报报表**：社保局申报表、公积金汇缴书等
- **增减员**：新增 / 减员 / 补缴 / 基数调整
- **商业保险**：补充医疗 / 意外险等

### 13. 合同与法务

- **合同模板**：劳动合同、保密协议、竞业限制等
- **合同管理**：签署、续签提醒（到期前 N 天）、变更、解除
- **劳动争议**：纠纷登记、处理记录、结果归档
- **法务文档**：员工手册、制度公告、确认签收

### 14. 审批工作流

- **流程引擎**：拖拽式流程设计器，条件分支，审批人设置
- **审批节点**：一人审批 / 会签 / 或签 / 依次审批
- **委托审批**：指定代理人、代理时间范围
- **催办与超时**：超时自动提醒、升级审批
- **审批记录**：审批轨迹完整日志、已办 / 待办 / 抄送
- **移动端审批**：适配企业微信 / 钉钉 / 飞书

### 15. 报表与看板

- **人力资源看板**：人数统计、入离职趋势、年龄/工龄/学历分布
- **薪酬分析**：薪酬结构、薪酬带宽、同岗薪酬对比
- **考勤报表**：出勤率、加班统计、请假分布
- **绩效分布**：绩效等级分布、部门对比
- **招聘分析**：招聘漏斗、渠道 ROI、平均招聘周期
- **自定义报表**：拖拽字段、条件筛选、图表类型、定时发送

### 16. 系统管理

- **用户管理**：账号、角色分配、启用/禁用
- **角色权限**：RBAC，功能权限 + 数据权限（按部门 / 按岗位）
- **操作日志**：关键操作记录、可追溯
- **系统配置**：参数配置（考勤规则 / 薪资项 / 审批流程等）
- **消息通知**：站内信、邮件、企微/钉钉推送
- **导入导出**：Excel 批量导入导出，数据校验
- **开放接口**：对接考勤机、银行、税务、第三方招聘平台

## 三、技术栈（已确认）

| 层 | 选定方案 | 说明 |
|---|---------|------|
| **后端** | **Spring Boot 4.1 + JDK 25 (LTS)** | 生态成熟，HR 领域案例丰富 |
| **前端** | **Vue 3 + Element Plus** | 上手快，中文文档好，后台组件覆盖全面 |
| **数据库** | **PostgreSQL + Redis** | PG 复杂查询能力强，Redis 做缓存/锁 |
| **架构** | **Monorepo + 单体模块化**（一期） | 前后端同仓库，按模块分包 |
| **部署** | **Docker Compose**（MVP） | 一期简化运维，后续上 K8s |
| **文件存储** | **MinIO** | 私有化部署，兼容 S3 API |
| **消息队列** | **RabbitMQ**（推荐） | 见下方详细对比分析 |
| **定时任务** | **XXL-JOB** | 薪资计算、考勤汇总等 |
| **审批引擎** | **自研轻量引擎** | 贴合 HR 审批场景，避免 BPMN 过重 |

### 技术栈讨论（决策过程记录）

> 以下为各层候选方案的对比分析，**粗体**为倾向性推荐，最终取决于团队技术储备和项目阶段。

### 3.1 后端框架

| 方案 | 优势 | 劣势 | 适合场景 |
|------|------|------|----------|
| **Spring Boot (Java)** | 生态最成熟，HR 领域案例多，事务/安全/批处理框架完善，招人容易 | 启动慢，配置重，JVM 内存占用高 | 大型企业、国企、对稳定性要求极高的场景 |
| **Go + Gin / Hertz** | 编译快，单体性能好，部署简单（单二进制），内存占用低 | 泛型/ORM 生态不如 Java，复杂 CRUD 模版代码略多 | 高并发 SaaS、云原生部署、中小团队 |
| **Node.js + NestJS** | 前后端同语言，开发效率高，TypeScript 全栈 | 计算密集型弱，回调链路复杂时排查困难 | 创业团队、快速原型、前端为主的小团队 |

> **HR 系统的特点是 CRUD 密集 + 复杂报表 + 审批流 + 定时任务**，对极致并发要求不高。
> 如果团队 Java 背景 → **Spring Boot**；如果团队 Go 背景 / 追求部署简洁 → **Go**。

### 3.2 前端框架

| 方案 | 优势 | 劣势 |
|------|------|------|
| **React + Ant Design Pro** | 组件生态最全，ProTable / ProForm 非常适合后台 CRUD，社区活跃 | 上手曲线略高，版本迭代快 |
| **Vue 3 + Element Plus** | 上手简单，中文文档友好，Element Plus 后台组件覆盖全面 | 大型项目状态管理略弱于 React |
| **Vue 3 + Naive UI** | TypeScript 支持好，按需加载性能佳，组件质量高 | 生态小于 Element Plus |

> HR 系统后台管理为主，**两个框架都能胜任**。关键看团队主力技术栈。

### 3.3 数据库

| 方案 | 适用场景 |
|------|----------|
| **PostgreSQL** ✅ 强烈推荐 | HR 数据关系复杂（多对多组织树、薪资历史、考勤明细），PG 的 CTE、窗口函数、JSONB、分区表正好应对复杂报表和历史数据查询 |
| MySQL | 如果团队运维只熟悉 MySQL 也可，但 PG 在复杂查询上明显更强 |

- **缓存**：Redis（分布式锁、会话、考勤缓存）
- **时序考量**：考勤打卡记录量大，可考虑 **TimescaleDB**（PG 扩展）或定时归档

### 3.4 架构风格

| 方案 | 适合阶段 |
|------|----------|
| **单体 + 模块化分层** ✅ MVP | 一期开发最快，部署最简单，后期按模块拆微服务 |
| 微服务 (Spring Cloud / go-kit) | 二期 / 三期，当模块间独立部署需求明确后再拆 |

> **建议**：一期单体优先，目录结构按模块分包（module-per-package），后续微服务化只需抽模块出去。

### 3.5 审批引擎

| 方案 | 优势 | 劣势 |
|------|------|------|
| **自研轻量引擎** ✅ MVP | 完全可控，贴合 HR 审批场景（会签/或签/条件分支），无外部依赖 | 需要开发投入，复杂 BPMN 场景需要持续打磨 |
| Camunda / Flowable | BPMN 2.0 标准，流程图设计器现成，社区成熟 | 太重，HR 审批大部分是简单审批链，BPMN 建模反而成本高 |
| 自研 + 状态机 | 轻量，适合固定审批流 | 灵活性不够，动态流程配置困难 |

> **建议**：MVP 自研一个轻量审批引擎（节点 + 条件 + 审批人解析），满足 90% HR 场景；
> 如果未来需要复杂流程编排（合同审批、采购审批等），再考虑集成 Camunda。

### 3.6 其他组件

| 层 | 推荐方案 | 说明 |

### 3.7 消息队列选型分析

> HR 系统中消息队列的使用场景：异步通知（审批待办推送、工资单生成完成）、模块解耦（考勤数据→薪资计算）、定时任务分发、操作日志异步写入。

| 维度 | Redis Stream | RabbitMQ | Kafka |
|------|-------------|----------|-------|
| **本质** | Redis 的数据类型之一 | 专业消息中间件 | 分布式事件流平台 |
| **定位** | 轻量消息通道 | 企业级消息代理 | 高吞吐事件管道 |
| **消息模型** | 消费组 + 消费者 | Exchange + Queue + Binding | Topic + Partition + Consumer Group |
| **持久化** | RDB / AOF（为缓存设计，非消息优先）| 消息确认后删除或保留 | 按配置保留（默认7天），可重放 |
| **吞吐量** | 中等（单线程模型，受限于 Redis 主线程）| 数万/秒 | 百万/秒 |
| **可靠性** | ACK 机制 + PEL（待处理列表），宕机可能丢少量数据 | AMQP 协议，Publisher Confirm + 消费 ACK，**可靠性最高** | 高可用 + ISR 副本机制，可靠性高 |
| **延迟** | 微秒级（内存操作）| 毫秒级 | 毫秒级（批量攒消息）|
| **消息回溯** | 不支持（消费后即删除） | 不支持（消费确认即删除）| **支持**（按 offset 回溯重放）|
| **死信队列** | 需手动实现 | **内置 DLX**（死信交换机） | 需手动实现 |
| **延迟消息** | 不原生支持（可通过 ZSET 近似实现）| **内置**（插件或 DLX 实现） | 不原生支持 |
| **运维成本** | 极低（复用已有 Redis） | 低（Erlang 写的，稳定，Java 客户端成熟）| **高**（ZooKeeper / KRaft，需专职运维）|
| **与 Spring Boot 集成** | Spring Data Redis 支持 | **Spring AMQP / RabbitTemplate 开箱即用** | Spring Kafka 支持 |
| **监控** | Redis 自带 MONITOR / INFO | **管理控制台**（队列深度、消费速率、连接数） | 需配套 Kafka Manager / Cruise Control |

#### HR 系统消息场景清单

| 场景 | 频次 | 要求 | 推荐方案 |
|------|------|------|----------|
| 审批待办通知（企微/钉钉/邮件推送）| 高频、短消息 | 低延迟、不丢消息 | RabbitMQ |
| 考勤打卡数据异步处理 | 高频（全员打上下班卡） | 顺序性要求低，吞吐要求中等 | RabbitMQ |
| 薪资计算任务分发 | 低频（月度）| 可靠性优先，不要重复/丢失 | RabbitMQ |
| 工资单生成 + 推送通知 | 低频（月度批量） | 事务性，确保全成功 | RabbitMQ |
| 合同到期批量提醒 | 低频（每日扫描） | 不关键 | Redis Stream 甚至 Spring Scheduler 即可 |
| 操作日志异步写入 | 高频 | 可以接受少量丢失 | Redis Stream |
| **预留**：未来大数据分析（考勤趋势、人力成本分析） | - | 高吞吐、支持回溯 | Kafka |

#### 结论

```
MVP      Redis Stream  ✅  够用（操作日志 + 简单通知）
正式上   RabbitMQ     ✅✅  推荐，HR 场景的完美匹配
大数据   +Kafka         可选，三四期再考虑
```

**推荐理由**：
1. **HR 系统的消息特征是"可靠小消息、中等吞吐"** —— 这正是 RabbitMQ 的设计目标
2. Spring Boot + RabbitMQ 集成度极高（`RabbitTemplate`、`@RabbitListener`、死信队列声明式配置）
3. 内置延迟消息（`rabbitmq-delayed-message-exchange` 插件），HR 审批超时提醒开箱即用
4. 内置死信队列（DLX），薪资计算失败的消息自动进入死信人工排查
5. 管理控制台清晰，非开发人员（运维）也能看懂队列状态
6. Redis Stream 的问题：复用 Redis 实例会影响缓存性能；单独部署 Redis 又失去"复用"优势

> **所以纠正之前的推荐**：中大型项目直接上 **RabbitMQ**，小型 MVP 先 Redis Stream 过渡也可以，但建议一步到位。

### 3.8 仓库策略：Monorepo vs 多仓库

| 维度 | Monorepo（单仓库） | 多仓库（前后端分离） |
|------|-------------------|---------------------|
| **目录结构** | `hr/ ├── backend/ ├── frontend/ ├── docs/` | `hr-backend/` + `hr-frontend/` + `hr-docs/` |
| **API 变更协调** | **容易** — 前后端同仓库，PR 里一起改 | 需要跨仓库协调，API 变更需先发布 contract 包 |
| **CI/CD** | 一套 CI 搞定，但需区分路径触发 | 各自独立 CI，互不影响 |
| **部署** | 需在 CI 中区分构建任务 | 可独立部署，前后端发布节奏解耦 |
| **代码复用** | **容易** — 共享类型定义、API 契约 | 需要抽成 npm / jar 包跨仓库引用 |
| **权限管理** | 整套代码在同一仓库，无法细粒度控制 | 可分别控制前后端权限 |
| **开发初期** | **效率高** — API 频繁调整时原子提交 | 频繁发版 contract 包，摩擦成本高 |
| **适合团队** | 小团队（< 10 人）、全栈开发 | 大团队、前后端明确分工、微服务 |

**已确认：Monorepo（单仓库）** ✅
